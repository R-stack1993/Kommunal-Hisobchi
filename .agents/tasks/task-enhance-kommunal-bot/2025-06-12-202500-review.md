# Kommunal Bot enhancement: i18n, PDF reports, SMS parsing, and secrets extraction

The Kommunal Hisobchi Telegram bot has been expanded from a single-language utility tracker into a multi-language system with PDF report generation, SMS forwarding/parsing, and externalized configuration. The original hardcoded bot token and database credentials were moved to `.env` via python-dotenv, a full Uzbek/Russian locale system was layered on top of all user-facing strings, monthly PDF reports can be generated on demand or auto-sent on schedule, and forwarded SMS messages are parsed via regex to extract meter readings.

Watch for: real credentials committed in FEAT-001.json task metadata (confirmed); `ADMIN_ID` crash on missing env var with unguarded `int()` cast (confirmed); SMS reading saved without validation against previous value — lets users record a lower reading than the last saved one, bypassing the manual-entry check (confirmed); no authorization check on house_id ownership in callback handlers (likely).

## High-level view

The secrets extraction achieves its goal — `.env` is gitignored and code reads from environment — but the task metadata file (`FEAT-001.json`) committed to the repository contains the real bot token and database password in its `steps` field. The credentials are now in git history regardless of whether `.env` itself is tracked.

The i18n layer uses a dict-per-language approach with a `get_text(key, lang, **kwargs)` dispatcher. Every handler fetches the user's language from the database before responding, which combined with the ban-check middleware means two separate DB round-trips before any handler logic runs.

PDF generation delegates to reportlab. The `get_monthly_report_data` function uses `MIN`/`MAX` aggregation on reading values rather than chronological ordering — a distinction that could produce incorrect reports if readings ever become non-monotonic.

SMS parsing extracts readings via regex then feeds them into the save path, but skips the `current < previous` validation that the manual entry flow enforces. A malformed or old SMS could corrupt a user's reading history.

7. **Double DB round-trip per request** — Every message and callback triggers `check_if_banned` (middleware) plus `get_user_language` in the handler, each opening a separate session. Consider combining into a single query or caching per-request.

The bot token should be revoked via BotFather and the database password rotated.

## Startup crash path

In `main.py`:

```python
ADMIN_ID = int(os.getenv('ADMIN_ID'))
```

If `ADMIN_ID` is not set, `os.getenv` returns `None`, and `int(None)` raises `TypeError`. The same pattern exists for `BOT_TOKEN` — `Bot(token=None)` would also fail — but the `ADMIN_ID` cast produces the most confusing traceback.

In `database.py`, `DATABASE_URL = os.getenv('DATABASE_URL')` feeds `None` into `create_async_engine(None)` at import time, producing an opaque SQLAlchemy error before the event loop starts.

## SMS reading bypass of validation

The manual reading flow in `process_current_reading` explicitly rejects values lower than the last stored reading:

```python
if last_record and current_reading < last_record.reading_value:
    await message.answer(get_text('reading_too_small', lang))
    return
```

The SMS path in `_save_sms_reading` does not perform this check:

```python
async def _save_sms_reading(...):
    reading_value = sms_data.get('reading')
    ...
    await save_new_reading(user_id, house_id, utility_type, reading_value)
```

If the regex extracts a value from an old SMS or a value in different units, it saves unconditionally. Subsequent manual entries will then fail the "current < previous" check against this corrupted value, effectively locking the user out of further entries for that house/utility until they manually clear history.

## House ownership in callbacks

Callback data like `pdfrep_house_{house_id}` and `selhouse_{utility_type}_{house_id}` are constructed with the user's own house IDs, but when the callback fires, the handler does:

```python
house_id = int(callback.data.split("_")[2])
house = await get_house_by_id(house_id)
```

`get_house_by_id` queries by `House.id` alone without filtering by `telegram_id`. A user who forges a callback payload with another user's `house_id` would get that house's data returned. For the PDF report path, this means they could see another user's readings and costs. For the reading-entry path (`selhouse_`), they could write readings into another user's house.

The Telegram API places no restriction on the callback payload string, and bot frameworks have been exploited this way before. Fix: add a `telegram_id` filter to `get_house_by_id`, or verify ownership in each callback handler.

## Auto PDF report: no rate limiting

`send_auto_pdf_reports` iterates over all users and all houses without any `asyncio.sleep` between API calls:

```python
for user_id in users:
    ...
    for house in houses:
        for utility_type in ["elektr", "gaz"]:
            ...
            await bot.send_document(...)
```

At scale this will trigger Telegram's rate limits (30 messages/second to different users). The broadcast function includes `await asyncio.sleep(0.05)` between sends, but the PDF auto-report function does not — an inconsistency that will cause `RetryAfter` exceptions once the user base grows.

## Locale filter brittleness

Message handlers use `F.text.in_([...])` to match button presses, with the list hardcoded at import time for each supported language:

```python
@dp.message(F.chat.type == "private", F.text.in_([
    get_text('btn_elektr', 'uz'), get_text('btn_elektr', 'ru')
]))
```

Adding a third language requires updating every handler's filter list. This is a maintainability concern, not a current bug.
