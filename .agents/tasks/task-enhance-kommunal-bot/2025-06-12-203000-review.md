# Kommunal Bot enhancement: i18n, PDF reports, SMS parsing, and secrets extraction (v2)

The five issues flagged in v1 that required code fixes have all been addressed. Env var validation now raises `RuntimeError` at startup for missing `BOT_TOKEN`, `ADMIN_ID`, and `DATABASE_URL`. The SMS reading path validates `current < previous`. House ownership is verified via `telegram_id` parameter in `get_house_by_id` across all callback handlers. Monthly report data uses chronological ordering. Rate-limiting is present in `send_auto_pdf_reports`. Credentials in FEAT-001.json are redacted.

Watch for: SMS cost calculation always yields zero in the fallback path because `get_readings_context` is called *after* the reading is saved â€” making the just-saved record the "last record" (confirmed); `_save_sms_reading` saves the reading before verifying house ownership, so a forged callback would insert orphaned data and then crash on `house.name` (likely).

## High-level view

The house ownership fix is thorough â€” every `get_house_by_id` call passes `telegram_id`, and handlers check for `None` before proceeding. The single gap is `_save_sms_reading`, which calls `save_new_reading` before the ownership-verified `get_house_by_id`, so the write happens unconditionally even if ownership would fail.

The SMS reading validation mirrors the manual-entry check, but the cost-display logic after the save has a sequencing bug: it re-fetches `get_readings_context` post-save, which now returns the just-saved reading as `last_record`, making the subtraction always zero when `sms_data['usage']` is absent.

## SMS cost calculation sequencing

The `_save_sms_reading` function has two code paths for displaying cost in the confirmation message:

```python
await save_new_reading(user_id, house_id, utility_type, reading_value)

house = await get_house_by_id(house_id, telegram_id=user_id)
# ...
usage = sms_data.get('usage')
if usage:
    cost, _ = calculate_tiered_cost(usage, utility_type)
else:
    last_record, _, _ = await get_readings_context(user_id, house_id, utility_type)
    if last_record and last_record.reading_value < reading_value:
        calc_usage = reading_value - last_record.reading_value
        cost, _ = calculate_tiered_cost(calc_usage, utility_type)
    else:
        cost = 0
```

When the SMS contains an explicit `usage` value (from the regex), cost is calculated correctly. When it doesn't (common â€” many SMS formats only include the current reading), the fallback queries `get_readings_context` to find the previous reading and compute usage as the difference. But `save_new_reading` was already called above, so `get_readings_context` returns the record just saved. `last_record.reading_value` equals `reading_value`, the `<` comparison is False, and cost is always 0.

The fix: the function already fetches `last_record` at the top for the validation check. Store that reference and reuse it for cost calculation instead of re-fetching after the save.

## Save-before-verify in SMS path

The main callback handlers (`selhouse_`, `chart_`, `pdfrep_house_`) all follow the pattern: verify ownership, abort if None, then proceed. `_save_sms_reading` reverses this:

```python
await save_new_reading(user_id, house_id, utility_type, reading_value)  # Write first
house = await get_house_by_id(house_id, telegram_id=user_id)            # Check ownership second
# house could be None here â€” house.name would crash
```

In practice this only triggers via a forged `sms_house_{id}` callback, since the bot only presents the user's own houses. But the pattern is inconsistent with the rest of the codebase and would crash rather than fail gracefully.
